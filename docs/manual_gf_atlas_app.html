🗺️ 1. Visão Geral e Importância

O GF Atlas App é a fundação estrutural do ecossistema digital da GREENFARMS. Seu papel é criar uma camada organizacional centralizada que conecta quem pode acessar o quê dentro do ERPNext, de forma rastreável, hierárquica e gerenciável.

Toda organização com múltiplos sistemas, relatórios, dashboards, documentos e aplicações enfrenta o mesmo desafio: como organizar o acesso ao conhecimento de forma eficiente? O ERPNext oferece controle de permissões por Role, mas esse controle é binário — o usuário tem ou não tem acesso a um DocType inteiro.

O GF Atlas resolve um problema mais fino: catálogo de conteúdo com controle de acesso por grupo. Em vez de gerenciar permissões individualmente para cada relatório ou página, você agrupa conteúdos em categorias (Content Groups) e define quem acessa cada categoria (Access Groups). O conteúdo herda automaticamente o grupo de acesso padrão de sua categoria.

O que o GF Atlas App entrega:

  • Catálogo centralizado de todos os conteúdos digitais da empresa (relatórios, dashboards, PDFs, automações, APIs, links externos...)
  • Grupos de acesso com permissões granulares por usuário (visualizar, buscar, criar, editar, deletar, exportar, imprimir)
  • Hierarquia de conteúdo em árvore — categorias e subcategorias organizadas por área
  • Herança automática de grupo de acesso: o conteúdo herda o grupo padrão de sua categoria, mas pode ter exceções
  • Rastreabilidade — cada conteúdo tem responsável, status, tipo, tags e rota
  • Extensibilidade do DocType Web Page, adicionando dois blocos HTML auxiliares para uso no Knowledge Hub
✅ Resultado prático: Com o GF Atlas instalado, qualquer portal interno, Knowledge Hub ou painel de navegação da GREENFARMS pode consultar qual conteúdo um determinado usuário tem permissão de ver, usando apenas dois parâmetros: o grupo de acesso do usuário e o registro do conteúdo.

🏗️ 2. Arquitetura do App

O GF Atlas App é um aplicativo Frappe customizado instalado como dependência no ERPNext. Ele é composto por 4 DocTypes, 3 Roles, um script de instalação automática e 2 campos custom no DocType nativo Web Page.

🏷️ 3. Roles e Permissões

O GF Atlas App cria automaticamente 3 Roles durante a instalação. Elas controlam o acesso aos DocTypes do próprio app (não são Roles do ERPNext padrão).

Role Perfil Leitura Escrita Criação Deleção Export/Print Share
GF Manager Gestores e administradores do atlas ✅✅✅✅✅✅
GF Editor Responsáveis por cadastrar e manter conteúdos ✅✅✅❌✅❌
GF Viewer Usuários que apenas consultam o catálogo ✅❌❌❌✅❌
ℹ️ Importante: Estas Roles controlam quem pode gerenciar o atlas em si (os DocTypes GF Access Group, GF Content Group, GF Content Registry). O controle de acesso ao conteúdo em si (relatórios, dashboards, etc.) é feito pelos grupos de acesso cadastrados no app — a Role sozinha não dá acesso ao conteúdo referenciado.

Como atribuir as Roles

Após a instalação, acesse ERPNext → Configurações → Usuário, abra o usuário desejado, vá na aba Roles e adicione a Role correspondente.

Exemplo de atribuição

  • Analista de TI responsável pela manutenção do catálogo → GF Editor
  • Gerente de Sistemas → GF Manager
  • Colaboradores que consultam o Knowledge Hub → GF Viewer

🔑 4. DocType: GF Access Group (Grupo de Acesso)

O GF Access Group é o coração do controle de acesso do GF Atlas. Ele define quem é em termos de perfil de acesso — não apenas através de uma Role do ERPNext, mas por meio de um grupo com identidade própria, flags comportamentais e uma lista explícita de usuários com permissões individuais.

Cada grupo representa um perfil de acesso da empresa: pode ser um departamento, um tipo de colaborador (interno, externo, consultor), uma combinação de acesso (público, restrito, executivo) ou qualquer segmentação relevante para o negócio.

💡 Diferença do ERPNext padrão: No ERPNext nativo, permissão = Role aplicada ao DocType. No GF Atlas, permissão = usuário listado em um Access Group com flags específicas (pode criar? pode deletar? está ativo?). Isso permite controlar acesso a conteúdos que nem são DocTypes — como URLs, relatórios externos e links.

👤 5. DocType: GF Access Group User (Child Table)

O GF Access Group User é uma Child Table — um DocType filho que vive dentro do GF Access Group, na aba de usuários. Cada linha representa um usuário e define suas permissões individuais dentro daquele grupo específico.

Esse design permite permissões granulares por usuário dentro do mesmo grupo: dois usuários no mesmo grupo Financeiro podem ter capacidades diferentes — um pode exportar e imprimir, o outro apenas visualizar.

🗂️ 6. DocType: GF Content Group (Grupo de Conteúdo)

O GF Content Group é um DocType do tipo Tree (Árvore) — o que significa que seus registros se organizam em uma hierarquia pai-filho com profundidade ilimitada. É a espinha dorsal da organização de conteúdo do GF Atlas.

Pense nele como uma árvore de categorias: no topo ficam as grandes áreas da empresa (RH, Financeiro, TI, Produção…), e dentro de cada uma existem subcategorias cada vez mais específicas até chegar nas folhas, que são os grupos que efetivamente classificam conteúdos registrados.

💡 Tree DocType no Frappe: Internamente, Frappe usa o modelo Nested Set (NSM) para armazenar árvores. Cada nó tem campos lft, rgt e parent_group gerenciados automaticamente. Você não precisa se preocupar com isso — basta informar o pai ao criar um grupo.

📋 7. DocType: GF Content Registry (Registro de Conteúdo)

O GF Content Registry é o catálogo propriamente dito. Cada registro representa um item de conteúdo da empresa — pode ser um relatório, um dashboard, um PDF, uma automação, uma página web, uma API ou qualquer outro tipo de item digital que a empresa queira organizar e controlar.

O autoname segue o padrão GF-REG-.YYYY.-.#####, gerando IDs como GF-REG-2026-00001. Isso garante rastreabilidade e unicidade permanente.

7.3 Auto-preenchimento do Grupo de Acesso

Este é um dos comportamentos mais importantes do GF Atlas. Quando o usuário seleciona um content_group, o sistema busca automaticamente o default_access_group daquele grupo e preenche o campo access_group — tanto no cliente (via JavaScript) quanto no servidor (via Python no validate).

# Python — gf_content_registry.py
def validate(self):
    if self.content_group and not self.access_group:
        default_ag = frappe.db.get_value(
            "GF Content Group", self.content_group, "default_access_group"
        )
        if default_ag:
            self.access_group = default_ag
// JavaScript — gf_content_registry.js
frappe.ui.form.on("GF Content Registry", {
    content_group(frm) {
        if (frm.doc.content_group) {
            frappe.db.get_value("GF Content Group", frm.doc.content_group,
                "default_access_group", (r) => {
                    if (r && r.default_access_group && !frm.doc.access_group) {
                        frm.set_value("access_group", r.default_access_group);
                    }
            });
        }
    }
});
ℹ️ Dois níveis de proteção: O JavaScript age imediatamente ao selecionar o grupo (UX fluida). O Python age no validate, garantindo que mesmo um registro criado via API ou import tenha o access_group correto — desde que não tenha sido definido explicitamente.

8.1 Fluxo de Herança do Grupo de Acesso

1

Definição do Grupo de Acesso

O administrador cria um GF Access Group (ex.: financeiro) e adiciona usuários na Child Table, definindo as permissões de cada um.

2

Associação ao Grupo de Conteúdo

Um GF Content Group folha (ex.: financeiro_relatorios) define default_access_group = financeiro. Isso significa que todo conteúdo desta categoria pertencerá ao grupo Financeiro por padrão.

3

Cadastro do Conteúdo com Herança

Ao criar um GF Content Registry com content_group = financeiro_relatorios, o campo access_group é preenchido automaticamente com financeiro — sem intervenção manual.

4

Exceção pontual

Um relatório específico precisa ser acessível também pela Diretoria? Basta editar aquele registro e trocar o access_group para diretoria. Os demais itens do grupo ficam inalterados.

5

Consulta no portal

O portal ou Knowledge Hub consulta via API: "quais registros têm access_group igual ao grupo do usuário logado, estão active = 1 e têm status = ACTIVE?" — e exibe apenas o que aquele usuário pode ver.

8.2 Exemplo de Consulta via API Frappe

# Buscar conteúdos acessíveis para um grupo específico
frappe.get_all(
    "GF Content Registry",
    filters={
        "access_group": "financeiro",
        "status": "ACTIVE",
        "active": 1
    },
    fields=["name", "title", "item_type", "route_url", "icon", "color"],
    order_by="sort_order asc"
)

⚙️ 9. Processo de Instalação

A instalação do GF Atlas App é feita com dois comandos no servidor Frappe. Tudo que o app precisa para funcionar — roles, grupos de acesso, grupos de conteúdo e campos custom — é criado automaticamente pelo script install.py.

Comandos de Instalação

# 1. Obter o app a partir do repositório GitHub
bench get-app https://github.com/flexfernandes/ICHIS_ATLAS_APP

# 2. Instalar no site desejado
bench --site greenfarmsagro.frappe.cloud install-app ichis_atlas_app

# 3. Executar migrate (garante que os DocTypes são criados no banco)
bench --site greenfarmsagro.frappe.cloud migrate
⚠️ Frappe Cloud: No Frappe Cloud, a instalação é feita pelo painel administrativo em Sites → seu-site → Apps → Install App. Informe a URL do repositório GitHub. O migrate é executado automaticamente pela plataforma.

9.1 O que o after_install executa

Quando o Frappe detecta que o app foi instalado, chama automaticamente a função after_install() definida em hooks.py. Esta função orquestra 4 etapas em sequência:

1

create_roles()

Cria as 3 Roles do app (GF Manager, GF Editor, GF Viewer) caso não existam. Usa frappe.get_doc com ignore_permissions=True.

2

create_access_groups()

Cria os 37 grupos de acesso pré-definidos — todos os departamentos e perfis da GREENFARMS. Cada grupo é criado apenas se não existir (idempotente). Inclui: público, visitante, terceirizado, consultoria, cliente, fornecedor, engenharia, automação, produção, qualidade, manutenção, suprimentos, logística, financeiro, comercial, RH, marketing, TI, ERPNext, IA & automações, gerência, diretoria, administrador, jurídico, controladoria, auditoria, segurança do trabalho, P&D, laboratório, processos industriais, integrações, dashboards, web applications, documentações, projetos especiais, arquivo morto, obsoleto.

3

create_content_groups()

Cria a árvore de 270+ grupos de conteúdo hierárquicos representando toda a estrutura de conhecimento da empresa (RH, Financeiro, Operações, TI, Engenharia, Automações, BI, etc.). Cada grupo recebe seu default_access_group, ícone Lucide e cor hexadecimal.

4

create_web_page_custom_fields()

Adiciona dois campos do tipo HTML Editor ao DocType nativo Web Page do Frappe: auxiliary_block_1 e auxiliary_block_2. Ver seção 10 para detalhes.

# Trecho do install.py — função principal
def after_install():
    create_roles()
    create_access_groups()
    create_content_groups()
    create_web_page_custom_fields()
    frappe.db.commit()
    print("GF Atlas App: instalação concluída com sucesso.")
✅ Idempotência: Todas as funções verificam se o registro já existe antes de criar. Isso significa que o after_install pode ser executado novamente sem duplicar dados — útil em reinstalações ou atualizações.

🌐 10. Campos Custom no DocType Web Page

Durante a instalação, o GF Atlas estende o DocType nativo Web Page do Frappe com dois campos HTML Editor. Esses campos permitem que cada página do portal interno (como o Knowledge Hub) tenha blocos de HTML completamente customizáveis, independentes do conteúdo principal.

Custom Field Rótulo Tipo Posição
auxiliary_block_1 Auxiliary Block 1 HTML Editor Seção "Auxiliary Blocks" (ao final do formulário)
auxiliary_block_2 Auxiliary Block 2 HTML Editor Após auxiliary_block_1

Como são criados

# install.py — create_web_page_custom_fields()
def create_web_page_custom_fields():
    meta_fields = frappe.get_meta("Web Page").fields
    last_fieldname = meta_fields[-1].fieldname if meta_fields else ""

    custom_fields = [
        {"fieldname": "gf_sb_auxiliary_blocks", "label": "Auxiliary Blocks",
         "fieldtype": "Section Break", "insert_after": last_fieldname},
        {"fieldname": "auxiliary_block_1", "label": "Auxiliary Block 1",
         "fieldtype": "HTML Editor", "insert_after": "gf_sb_auxiliary_blocks"},
        {"fieldname": "auxiliary_block_2", "label": "Auxiliary Block 2",
         "fieldtype": "HTML Editor", "insert_after": "auxiliary_block_1"},
    ]
    for f in custom_fields:
        field_id = f"Web Page-{f['fieldname']}"
        if not frappe.db.exists("Custom Field", field_id):
            frappe.get_doc({"doctype": "Custom Field", "dt": "Web Page", **f}).insert(ignore_permissions=True)

Como usar no Knowledge Hub

Após a instalação, abra qualquer Web Page no ERPNext. Role até o final do formulário — a seção Auxiliary Blocks estará disponível. Escreva HTML diretamente nos editores.

No template Jinja da página ou via API Python, acesse os campos assim:

# Via API Python (template Jinja no Web Page)
{{ doc.auxiliary_block_1 }}
{{ doc.auxiliary_block_2 }}

# Via frappe.db (script Python)
doc = frappe.get_doc("Web Page", "knowledge-hub")
bloco_1 = doc.auxiliary_block_1
bloco_2 = doc.auxiliary_block_2

# Via API REST
GET /api/resource/Web Page/knowledge-hub
# Retorna auxiliary_block_1 e auxiliary_block_2 no JSON
💡 Caso de uso prático: O Knowledge Hub da GREENFARMS usa auxiliary_block_1 para renderizar um painel de acesso rápido aos conteúdos mais visitados, e auxiliary_block_2 para um bloco de avisos e comunicados. Tudo gerenciado diretamente pelo editor do ERPNext, sem deploy de código.

🎯 11. Casos Práticos de Uso

Cenário 1 — Cadastrar um Relatório de RH no Catálogo

A analista de RH quer registrar o relatório de absenteísmo mensal para que os gestores de RH possam encontrá-lo facilmente no portal.

1

Acesse GF Content Registry → Novo

Menu ERPNext → GF Atlas App → GF Content Registry → botão + Novo.

2

Preencha Título e Tipo

title = "Relatório de Absenteísmo Mensal" | item_type = REPORT

3

Selecione o Grupo de Conteúdo

content_group = rh_relatorios_pessoas → o campo access_group é preenchido automaticamente com rh.

4

Informe a Rota e o Responsável

route_url = /app/query-report/Employee Analytics | responsible_user = rh@greenfarms.com

5

Salve — o ID é gerado automaticamente

O registro recebe o nome GF-REG-2026-00042 e fica disponível para consulta via API e portal.

Cenário 2 — Criar um Novo Grupo de Conteúdo para uma Nova Área

A empresa cria o departamento de Inovação e precisa de uma categoria de conteúdo dedicada na árvore.

1

Crie o Grupo de Acesso para Inovação

GF Access Group → Novo: internal_name = inovacao | display_name = Inovação | is_internal = true. Adicione os usuários na aba Usuários.

2

Crie o Grupo Pai na Árvore

GF Content Group → Novo: internal_name = inovacao | display_name = Inovação | is_group = true | parent_group em branco | default_access_group = inovacao | icon = lightbulb | color = #7c3aed.

3

Crie as Subcategorias Filhas

Ex.: inovacao_projetos com parent_group = inovacao | is_group = false. Repita para cada subcategoria necessária.

Cenário 3 — Adicionar um Novo Colaborador ao Grupo de TI

Um novo analista de TI foi contratado e precisa ter acesso ao conteúdo do departamento.

1

Garanta que o usuário existe no ERPNext

Configurações → Usuário → verifique que o e-mail do novo colaborador já está cadastrado e ativo.

2

Abra o GF Access Group "ti"

GF Access Group → abra o registro ti → aba Usuários.

3

Adicione uma nova linha na Child Table

Clique em Adicionar Linha, selecione o usuário. O nome é preenchido automaticamente. Configure as permissões conforme o perfil do colaborador e salve.

4

Acesso imediato

No próximo login, o portal consultará o access_group do usuário e exibirá automaticamente todos os conteúdos de TI que estejam ativos.

❓ 13. Perguntas Frequentes

O GF Atlas substitui o controle de permissões do ERPNext?

Não. O GF Atlas é complementar. O ERPNext controla quem pode acessar cada DocType. O GF Atlas controla quais itens de conteúdo catalogados (relatórios, links, PDFs…) cada grupo de usuários pode ver em portais e dashboards personalizados. Os dois sistemas coexistem.

O que acontece se eu reinstalar o app?

O after_install é idempotente: verifica se cada registro já existe antes de criar. Roles, grupos de acesso e grupos de conteúdo existentes não são sobrescritos. Os Custom Fields também são verificados antes de inserir.

Posso adicionar novos campos aos DocTypes do GF Atlas?

Sim, via Custom Fields do ERPNext. Evite alterar diretamente os arquivos JSON do repositório, pois isso pode causar conflitos em atualizações futuras. Use a interface de Customização de Formulário do ERPNext.

Como consulto todos os conteúdos de um usuário específico?

Determine o grupo de acesso do usuário (via GF Access Group User), depois filtre GF Content Registry por esse access_group, status = ACTIVE e active = 1. Ver exemplo de consulta na seção 8.2.

Um conteúdo pode pertencer a mais de um grupo de acesso?

No modelo atual, cada GF Content Registry tem um único access_group. Para que um conteúdo seja acessível por múltiplos grupos, crie registros duplicados com o mesmo route_url e diferentes access_group. Uma futura versão pode adicionar uma child table de grupos de acesso.

Os ícones Lucide funcionam automaticamente?

O campo icon armazena apenas o nome do ícone (ex.: users, bar-chart-2). A renderização visual depende do template do portal. Para ativar, inclua a biblioteca Lucide Icons no HTML e chame lucide.createIcons() após o carregamento da página.
